# Investigating Products

프로덕트를 하나씩 뜯어보며 기록하는 분석 아카이브입니다.

평소에 즐겨 쓰거나 궁금했던 서비스를 골라, 비즈니스 모델부터 사용자 경험까지 직접 데이터를 모아 분석하고 개선안을 제안합니다. 잘 만든 서비스를 관찰하는 것이 가장 좋은 공부라고 믿어서, 한 번에 하나의 프로덕트를 깊게 파는 것을 원칙으로 합니다.

## 분석 아카이브

| # | 프로덕트 | 한 줄 요약 | 링크 |
|---|---|---|---|
| 01 | 리멤버 (Remember) | 옳은 방향(Platform 확장)으로 가고 있으나 실행이 코어(Product)를 훼손 중. 균형을 되찾는 '스카웃 라벨 가시화' 제안 | [보러 가기](https://gaoningmomo.github.io/investigating_products/remember/) |

## 각 분석의 구성

모든 분석은 같은 뼈대를 따릅니다.

1. **Overview**: 비즈니스 모델 캔버스로 구조 파악
2. **서비스 개요와 정체성**: 시장 내 포지션, 서비스의 본질 정의
3. **핵심 문제 진단**: 데이터 기반 증상 분석 (리뷰 크롤링, 키워드 태깅 등)
4. **개선안 도출 과정**: 브레인스토밍부터 우선순위 결정까지의 여정
5. **개선안 심층 설계**: 1순위 개선안의 구체적 설계와 성공 지표
6. **가설·검증 계획**: 팩트와 가설의 분리, 검증 방법 명시
7. **회고**: 프로젝트에서 얻은 배움

## 원칙

- 확신보다 가설. 검증 안 된 주장은 가설이라고 명시한다
- 프레임워크는 도구지 답이 아니다. 질문에 맞는 프레임을 고른다
- 정량 데이터와 직접 사용 경험을 함께 본다
- 좁게, 깊게. 다 하려다 아무것도 안 되는 것보다 하나를 깊이 판다
